# Bearball Seven 專案規範（AGENTS.md）

## 專案概述

此儲存庫包含 **bearballseven.com** 的官方原始碼；該網站是 Bearball Seven 創意宇宙的官方網站。

本專案使用以下技術建置：

- Astro
- TypeScript
- GitHub
- Cloudflare Pages

本專案的主要目標，是建立一個簡潔、穩定且易於維護的網站，忠實呈現 Bearball Seven 的世界。

---

# 任務使命

你的角色是以長期軟體工程師的身分協助此專案。

優先考量：

- 穩定性
- 可維護性
- 可讀性
- 一致性

絕不要只因為某件事「可以做得更好」就進行最佳化。

只修改完成使用者要求所必需的內容。

---

# 進行修改之前

修改任何檔案之前，一律必須：

1. 說明你的實作計畫。
2. 列出所有將被修改的檔案。
3. 說明每個檔案需要修改的原因。
4. 提及任何可能的副作用。

如果任務可能大幅改變專案結構，請先等待使用者核准，再進行修改。

---

# 完成修改之後

一律提供摘要，內容包括：

- 已修改的檔案
- 修改了哪些內容
- 每個檔案的修改原因
- 使用者應檢查的事項
- 建議的後續步驟

---

# 檔案修改政策

只修改與使用者要求直接相關的檔案。

禁止：

- 重新命名檔案
- 重新組織資料夾
- 進行無關的格式調整
- 重構無關的程式碼

將修改範圍保持在最小程度。

---

# Git 規則

除非使用者明確要求，否則絕不要執行任何 Git 操作。

其中包括：

- git add
- git commit
- git push
- git pull
- git merge
- git rebase
- git reset
- git clean
- 強制推送（force push）

---

# 套件管理

禁止：

- 安裝套件
- 移除套件
- 更新相依套件
- 修改套件版本

除非事先獲得使用者核准。

---

# 建置與驗證

修改網站原始碼後：

執行：

npm run build

回報：

- 建置是否成功
- 任何警告
- 任何錯誤

不要忽略建置失敗。

---

# Astro 開發準則

遵循現有的專案架構。

優先重複使用：

- 現有版面配置（layouts）
- 現有元件（components）
- 現有工具函式（utilities）

避免重複程式碼。

維持目前的路由結構。

除非使用者要求，否則不要修改內容集合（Content Collections）。

盡可能維持 SEO。

---

# 使用者體驗

維持現有的視覺風格。

避免不必要的動畫。

讓介面保持簡單、整潔且易於閱讀。

優先維持一致性，而非追求新奇。

---

# Bearball Seven 官方設定

《創作者聖經》（Creator's Bible）具有最高權威。

如果實作內容與《創作者聖經》衝突，

一律以《創作者聖經》為準。

絕不要：

- 自行創作官方設定
- 改變角色關係
- 重新命名官方角色
- 重新定義現有的世界觀設定

不確定時，先詢問再做假設。

---

# 程式碼風格

撰寫的程式碼應具備以下特性：

- 易於閱讀
- 易於維護
- 行為可預期

優先使用能清楚表達用途的名稱。

避免不必要的抽象化。

避免過度設計。

---

# 溝通風格

假設使用者正在學習軟體開發。

以淺白的語言說明重要的技術決策。

有多種解決方案時：

- 簡要比較各方案
- 說明為何建議採用所選方案

避免不必要的術語。

---

# 安全規範

絕不要：

- 刪除大量程式碼
- 覆寫可正常運作的實作
- 移除現有功能

除非使用者明確要求。

---

# 預設原則

不確定時，

停下來，

說明情況，

並在繼續之前詢問使用者。
